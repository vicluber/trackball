# Blackberry Trackball — ZMK Firmware

Standalone USB HID mouse using a Blackberry optical trackball wired to an nRF52840 (nice!nano v2).

## Wiring

| Trackball pin | nRF52840 pin |
|---------------|--------------|
| VCC           | VCC          |
| GND           | GND          |
| UP            | P0.20        |
| DOWN          | P0.22        |
| LEFT          | P1.15        |
| RIGHT         | P0.24        |
| BTN (click)   | P0.02        |

## What the driver does

`drivers/input/bb_trackball.c` is the piece that was missing from all existing
ZMK configurations for this hardware.

**The problem it solves:** The trackball's hall sensors hold the direction pins
at a voltage based on where the magnetic rollers physically stopped. A naive
approach that reads "is the pin currently LOW?" will always see *some* pin as
active and the cursor drifts forever.

**How it works:** Instead of reading pin levels, the driver registers a
GPIO interrupt on each of the four direction pins. The interrupt fires only on
a **falling edge** — the moment a pin transitions from HIGH to LOW. That
transition corresponds to a real physical movement of the ball. Each edge fires
a handler that calls `input_report_rel()` to emit one unit of REL_X or REL_Y
into Zephyr's input subsystem. ZMK's `zmk,input-listener` picks that up and
forwards it to the USB HID mouse report.

```
ball rolls → hall sensor pulses → falling edge interrupt
  → handler calls input_report_rel(REL_X, +1)
    → zmk,input-listener forwards to HID
      → PC sees mouse move right by 1 unit
```

The click button is handled separately by `zmk,kscan-gpio-direct` (level
detection is fine for a real button) and mapped to left click in the keymap.
